## Review loop (dual: Opus whole-branch + Fable security)

Opus: approve, no actionable findings — no-bypass rule grep-verified, retirement complete, bundle discipline (type-only main-thread imports, worker chunk) confirmed, D35/P1-guard byte-intact. One load-bearing invariant to know: `CompileResultsInbox` is single-key-per-`(turnId,kind)` — safe because the supervisor sequences check/full strictly, mirroring the verify inbox. Fable (security): trust boundary SOUND — ownership 404-never-403, Defense-4 cross-tenant verdict isolation, and verify-before-announce bound to the server-side projectId all hold with tests; forged green results bound to the attacker's own prefix. Five hardening findings, ALL fixed in `c29dc84` (+12 tests): **M1** per-project staged-bytes/prefix quotas + `sweepStaged` GC (uncommitted-staging disk-exhaustion DoS closed); **L2** per-prefix write mutex (bundle-cap TOCTOU); **L3** NUL/C0 rejection in the shared `isSafePath` (4xx not 500; also hardens SRS + handoff); **L4** capped `buildServer` default store; **L5** coordinator-level `compile:results` cross-tenant regression test. A fix-diff re-review (Opus) approved all five and empirically found a **data-loss race**: the disk `sweepStaged` could `rm` a prefix that `commit()`ed mid-sweep (stagedAt is PUT-time, so a slow commit after an old last-PUT was a genuine sweep target) — fixed in `f0b1cc0` (sweep re-checks `isCommitted` under the per-prefix lock, `commit()` takes the same lock; TDD-reproduced the manifest loss then closed; 626 tests). Corrected residuals (accepted for demo scope, ownership-gated, single-instance): (a) the staged caps are enforced against sibling state read OUTSIDE a project-wide lock, so a concurrent burst of N PUTs to N distinct sourceHashes can transiently overshoot by ~N× one bundle before the caps clamp (bounded by attacker in-flight concurrency, NOT the prefix-count cap as first written); lingers up to the 1 h sweep cutoff. (b) M1 caps only UNCOMMITTED footprint — no ceiling/GC on the COUNT of committed prefixes (pre-existing, tracked for a later story). (c) `withPrefixLock` is in-process — the single-instance-owns-the-artifact-root assumption (consistent with L2/M1) holds under Fly scale-to-zero; multi-instance would need an fs-level guard.
