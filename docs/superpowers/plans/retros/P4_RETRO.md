## Discoveries

- **CONSTITUTION-I CATCH (Task 2, corrected in Task 3b):** the recipe's _prose_ SDK field names were wrong — live indexer introspection showed `Transaction` is a GraphQL interface with `transactionResult` only on the `RegularTransaction` inline fragment, and the raw status enum is `SUCCESS`/`PARTIAL_SUCCESS`/`FAILURE` (not the SDK's mapped `SucceedEntirely`). Written from the prose it would have been a silent finality-query bug; the live devnet caught it. `sdk-recipe.md` Element 4 was corrected. **Standing rule for all indexer-touching code: verify field names against live introspection, never recipe prose.**
- **⚠️ Genesis seed `0x…04` is EMPTY on this devnet** — only `0x…01`–`0x…03` are funded (SPIKE-2's "seeds 01–04" overstated). Partition in use: 01 = SPIKE-1, 03 = SPIKE-2, 02 = P4. **P5's funding must use funded seeds and a clean partition** (server deploy wallet vs user dev wallet on distinct funded seeds).
- **Finality is one definition, two consumers:** the deploy `awaitFinality` and the deposit `DepositsStateReader.finalized` both derive from the same signal (indexer-served state = GRANDPA-finalized). Never a hardcoded `finalized:true` (the P3 I1 lesson) — the store's SC-021 gate stays live, proven end-to-end through the real query + store on the false path. The stricter node-finalized-head cross-check is a documented body-only hardening seam.
- **`getLatestGreenBuild` is real end-to-end now** (P1 store + migration 0005), so the deploy greenness gate reads real rows — the Phase-10 stub is fully retired.
- **Install side-effects P0's hardening surfaced:** the SDK install flipped the hoisted `@types/node` to 18.x (transitive via testkit's `@types/ssh2`) breaking `infra` typecheck → pinned `@types/node@26.1.1` in `infra/`; pnpm's allowlist ignored `classic-level`'s native build (add to `onlyBuiltDependencies` with justification if a devnet-gated test needs it, never lower the gate).

## Review loop (dual: Fable deploy-key/money + Opus whole-branch)

No Criticals — SC-031 key isolation (exactly-2 construction sites, name-scan + excise-and-assert, runtime canaries), finality-exactly-once (record-before-announce, memoised terminal arms, dead-defensive `reorged`), the SC-021 crediting gate (proven live through the real reader→query→store with db/ledger fakes that throw if touched), and the submission mutex all check out. Two Importants fixed: **I1 (money-critical)** `awaitFinality` called `queryFinality` with no try/catch, but the real adapter throws on a transient indexer blip AND on a finalized-but-no-address tx — both rejected out → the pipeline marked failed+retriable → a retry re-drove prove→submit → a SECOND on-chain deploy (real tDUST double-spend, certain in the no-address case). Fixed: transient throws are swallowed as `pending` within the bounded wait (degrades to the honest `timeout`, which keeps the txRef and never retry-redeploys); the finalized-but-no-address case maps to a distinct NON-retriable failure + loud log for ops reconcile. **I2** the buildDeploy/submit/funds-gate catches swallowed errors with zero server log (and mis-classified the not-wired submit as "node rejected") — fixed with an injectable `logError` that logs **`error.name` only** (SC-031-safe: a real SDK error can echo the deploy key in its message, so name-only preserves isolation and keeps the exposure grep valid; a canary test proves the sink never receives the key). Minors: M1 runtime guards on the decoded vault entries (bigint amount / 32-byte ref → loud `VaultModuleLoadError`); two vacuous `.catch`-only leak tests given `.rejects` guards.

Documented-and-carried (not code — pre-existing or owner-scoped): the submission mutex + in-flight set are **process-local** (single-instance-owns-the-deploy-wallet assumption; under Fly scale-out two instances would race the one wallet's UTXO state — P5 should add a boot-time single-instance assertion or a deploy-doc note); the pipeline's per-request replay memo is inert in production wiring (durable exactly-once rests on the registry `tx_ref` idempotency, which holds — only matters if requestId-driven client retries are ever added); most newly-installed `@midnight-ntwrk/*` pins are pre-installed for P5's body-only wiring and not yet statically imported (**P5 must confirm every pin is consumed** when it wires the real bodies). **Owner-gated round-trip checklist addition (Fable question):** probe whether a `ContractDeploy` intent can sit in the _guaranteed_ segment of a `PARTIAL_SUCCESS` tx (contract created on-chain while the pipeline reports retriable-failure → orphaned contract + retry → second deploy) — the recipe maps `PARTIAL_SUCCESS→failed` per the SDK's own criterion, but the guaranteed-segment interaction is unprobed on a single-node devnet.
