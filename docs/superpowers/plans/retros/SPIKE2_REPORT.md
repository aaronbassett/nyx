The injection point is a **supported SDK seam, not a hack**: `ledger-v8`'s `Transaction.prove(provider: ProvingProvider, costModel)` takes any `{check, prove}` callback pair, and `@midnight-ntwrk/zkir-v2` exports `provingProvider(keyMaterialProvider)` producing exactly that shape. The wallet SDK's own `wallet-sdk-prover-client` `WasmProver` is this same wiring (it is how official in-browser wallet proving works), so the approach is aligned with upstream architecture, not fighting it.

Caveats inside the YES:

- The **wallet's fee leg** (DUST spend) was proven by the wallet stack through its configured prover — the devnet proof server (`POST /prove` at 18:40:37, one call, correlating with the wasm run). The contract-circuit proof demonstrably did NOT touch the proof server (see Evidence §E). In production this split is exactly right: Lace proves its own wallet legs internally; the DApp side proves the contract circuit. A fully serverless path for the wallet leg also exists upstream (`WasmProver.asProvingProvider()` in `wallet-sdk-prover-client@1.2.2`, same zkir engine) but was not exercised.
- The PoC's wasm **compiler** (compact HEAD `c06961eb66`, codegen for runtime 0.18-rc) was NOT used for the accepted transactions; the contract was compiled with the native compact **0.31.1** toolchain (runtime 0.16.0), matching Nyx's pinned toolchain and the installed midnight-js 4.1.1 stack. In-browser proving does not require in-browser compiling.
- Runs were in Node (same wasm binaries the browser loads). The PoC already demonstrates this engine running in an actual browser; what the browser adds is memory/time pressure, not a different code path.

### What a submittable Midnight tx needs beyond zkir `prove` (from installed SDK source, not memory)

The zkir proof bytes are one callback's return value inside a larger pipeline, all of which the SDK provides:

1. **Unproven tx assembly** — `midnight-js-contracts` builds `Transaction<SignatureEnabled, PreProof, PreBinding>` from a `ContractCallPrototype` (transcripts + input/output alignments + communication-commitment randomness) via `Transaction.fromPartsRandomized(networkId, …)`; deploys wrap `new ContractDeploy(contractState)` (verifier keys inside `ContractState`) in an `Intent`.
2. **Proving** — `tx.prove(provingProvider, CostModel.initialCostModel())` (ledger-v8 wasm orchestrates per-circuit preimages → your `prove` callback returns proof bytes → wasm reassembles a `Transaction<S, Proof, PreBinding>`). **There is no lower-level "splice proof bytes into a tx" API** — the callback IS the interface; this is fine.
3. **Balancing + fees + signing + binding** — wallet-sdk facade: `balanceUnboundTransaction` (adds NIGHT inputs for the contract's `receiveUnshielded` claim + DUST fee spend) → `signRecipe` (BIP-340 Schnorr over the unshielded intents via the unshielded keystore) → `finalizeRecipe` (proves wallet legs, binds) → `FinalizedTransaction`.
4. **Serialization + submission** — handled by `wallet.submitTransaction(finalizedTx)` over the node WS relay (`ws://127.0.0.1:9944`); finalization observed via indexer `watchForTxData(txId)` (`transactions`/`contractAction` GraphQL on `:8088/api/v4/graphql`).

## Fallback decision

**Proof-server route proven — yes** (Evidence §C). Exact recipe: standard `midnight-js@4.1.1` provider stack with `httpClientProofProvider('http://127.0.0.1:6300', zkConfigProvider)`; note the modern proof server is NOT the legacy whole-tx `/prove-tx` — midnight-js drives the same `Transaction.prove` seam and POSTs each circuit's serialized preimage + client-supplied key material (`{proverKey, verifierKey, ir}` from the zkConfigProvider) to `/check` and `/prove` (`application/octet-stream`). This means **even the fallback requires shipping the contract's keys/IR from the client side** — the proof server holds only the built-in zswap/dust keys. Both routes therefore share the same artifact-distribution story (Nyx's R2 artifact prefix already carries exactly these files), and switching between in-browser and proof-server proving is a one-line `proofProvider` swap.

## Risks for P3 (dev-wallet / ceremony plan)

1. **Proving cost is real but bounded**: k=13 NyxtVault `deposit` proof = ~23–26 s in Node on this box, 4.5 KB proof; prover key 2.8 MB + IR + SRS 1.5 MB (k=13) must reach the client (or be keygen'd in ~16 s). Browser numbers will be of the same order (PoC measured 22–23 s for k=13 in its own harness). Budget UX accordingly (progress states, web worker, prefetch of key material from R2).
2. **Version pinning is the load-bearing constraint**: the working matrix is compact 0.31.1 (`ir-source[v2]`, runtime 0.16.0) + `zkir-v2@2.1.0` + `ledger-v8@8.1.0` + midnight-js 4.1.1 + wallet-sdk 1.0.0 + node 0.22.5/proof-server 8.1.0. The zkir wasm built from ledger-9.1.0-rc.3 happens to be byte-compatible for v2 IR, but a compiler upgrade that emits a newer IR version (the PoC's HEAD compiler does) or a node/ledger major bump can break the proof or key format silently — pin all three legs together and add a canary (prove+verify one known circuit) on any bump.
3. **The wallet leg still needs a prover**: balancing adds a DUST-spend proof, produced by whatever wallet is in play. Real Lace proves internally (fine); Nyx's own dev-wallet/ceremony must either point at the D37 proof server for its fee leg or wire `wallet-sdk-prover-client`'s `WasmProver` (same zkir engine; exists upstream, unexercised here). Do not assume "in-browser contract proof" removes the wallet's proving dependency.
4. **DUST registration must be part of any fresh-wallet onboarding** (P5 script and any ceremony that mints a throwaway wallet): transfer NIGHT → register → ~12 s accrual before the first fee-paying tx. Genesis seeds skip this but are shared global state on the devnet — concurrent agents/spikes should partition seeds or fund per-run child wallets.
5. **Key/artifact distribution**: BOTH proving routes require the client to supply `{proverKey, verifierKey, ir}` per circuit (the proof server holds only built-in circuit keys). Nyx's R2 artifact layout already matches `NodeZkConfigProvider`'s (`keys/<circuit>.prover|.verifier`, `zkir/<circuit>.bzkir`) — keep that contract; the browser needs a fetch-based KeyMaterialProvider over the same prefix (CORS/COEP per the R3 header config).
6. **`signData` dependency**: unshielded intents (deposit funding + dust registration) are signed via the unshielded keystore's BIP-340 `signData`. In the real ceremony that signature comes from Lace's connector — the same owner-gated real-Lace `signData` round-trip that US5 still has open gates this too.
7. **Concurrency**: per-wallet UTXO/nonce state means parallel submissions from the same seed can race (observed a transient `NIGHT=0` read mid-registration as UTXOs were respent/recreated). Serialize per-wallet, or use per-run wallets.
