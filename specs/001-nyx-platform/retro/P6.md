# Phase 6 Retrospective: User Story 3 — WebContainer preview + file-sync

## What Worked Well
- **The `WebContainerHandle` seam + `createRealHandle` adapter** made the whole phase unit-testable without a browser. A narrow, mockable interface (mount/spawn/fs/onServerReady/onError/teardown) that the real `@webcontainer/api` is adapted to; the real boot only runs under cross-origin isolation, so it stays owner-gated while all six modules test against a fake handle (54 unit tests). Same for the `PreviewBridge` WS seam.
- **Verify the real API + the POC before dispatching** (constitution I): read the installed `@webcontainer/api@1.6.4` `.d.ts` for the exact `WebContainer`/`WebContainerProcess`/`FileSystemTree` shapes, and mapped the proven boot sequence from the `webcontainer-lace` POC — so the agents built on facts, not memory of the WebContainer API.
- **Parallel component agents on disjoint files** (ws-client / boot / sync, then streams / env / artifacts) with the orchestrator running the integration typecheck at each commit gate.

## What Didn't Work
- **Scoped agent verification misses whole-package `tsc`.** Agents were told to skip whole-package `tsc` (it races concurrent writers) and verify via their own vitest (esbuild — no strict typecheck) + eslint. That let an `exactOptionalPropertyTypes` error slip (the sync test recorded `rm` options `boolean|undefined` into a `boolean` field) — caught only by the orchestrator's integration `tsc`. Lesson reaffirmed: the ORCHESTRATOR must run the full `lint && format:check && typecheck && test` at every integration gate; agent-green ≠ integration-green.
- **Hand-written seam files need lint+prettier, not just typecheck.** My `types.ts`/`real-handle.ts` typechecked but tripped `no-confusing-void-expression` (void expr from a shorthand arrow: `kill: () => proc.kill()`) + prettier. After that I pre-linted `env-file.ts` and it was clean first time.

## Workarounds & Solutions
- **`@webcontainer/api` process output** is `ReadableStream<string>` (already decoded — no TextDecoder); drain via a `getReader()` loop inside `try/finally` (always `releaseLock()`); `.exit: Promise<number>`. Drain the finite `npm install` (backpressure) but NOT the long-lived `npm run dev` (never exits).
- **Manifest DTO name** is `ManifestEntry` (schema `ManifestEntrySchema`), response `ProjectManifestResponse = ManifestEntry[]` — NOT the `ManifestRow` the plan guessed. `ContentHash` is a branded string; `FilePath` is not.
- **Shared `.env.local` (D10):** a single `ContainerEnv` (env-file.ts) merges keys so `contract:deployed` (`VITE_CONTRACT_ADDRESS`) and `artifacts:ready` (`VITE_ZK_CONFIG_BASE_URL`) never clobber. A `.env.local` change needs a dev-server RESTART (Vite reads env at startup) — the restart mechanic is injected.

## Patterns & Code
- `createPreview(handle, bridge, deps): PreviewController { start(); dispose() }` wires bridge events → `sync.applyWrite/Delete`, `handleContractDeployed`, `repointer.handleArtifactsReady`, `subscribeTakeover`, and `crashPolicy.crash` via `handle.onError` (D39's trigger); `start()` runs `runBootPipeline` (relaying each process via `streamProcessConsole`) then `sync.markMounted()`. `launchPreview` is the owner-gated real-boot entry. Barrel `container/index.ts` = `export *` over all 11 modules (no name collisions).
- VFS sync (D26/EC-14/FR-019): per-path promise-chain serialization (failure-tolerant), a pre-mount queue flushed by `markMounted()`, `isExcluded` dropping `node_modules/` + `.nyx/`.
- Crash policy (D39): `healthy → rebooted → terminal`; state advances BEFORE awaiting `reboot()` so a crash during an in-flight reboot is the terminal second crash; `reset()` re-arms.
