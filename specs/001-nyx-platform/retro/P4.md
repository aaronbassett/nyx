## Patterns & Code
<!-- Reusable patterns, helpful code snippets, architectural insights -->
- **Version model = project-wide monotonic counter.** A commit (turn-scoped batch OR one user edit) is a single `db.transaction`: `SELECT id FROM projects … FOR UPDATE` (serialize per-project allocation, D40 belt) → `SELECT COALESCE(MAX(version),0)+1 FROM project_file_versions` → per path UPSERT `project_files` (current, PK project_id,path) + INSERT `project_file_versions` (history). A rolled-back commit leaves NO history rows, so the version is NOT consumed — the next good commit reuses it (this is exactly what SC-026 asserts).
- **Atomic count-guarded quota INSERT** avoids a check-then-insert race: `INSERT INTO projects (…) SELECT $1,$2 WHERE (SELECT count(*) … WHERE owner=$1 AND deleted_at IS NULL) < $quota RETURNING …` — 0 rows returned ⇒ over quota ⇒ throw the named error.
- **Ownership denies with 404, not 403** (missing AND not-owned both 404) so existence never leaks (SC-027). `loadOwned(store, auth, id, reply)` centralizes it.
- **Named-error → HTTP status map** lives in one `handleStoreError`: FileTooLarge/ProjectQuota→413, CountQuota→409, RestoreWindowExpired→410, NotFound→404. Size checks use `Buffer.byteLength(content,"utf8")`, never string length; nothing is ever truncated.
- **Manifest** = `project_files` → `(path, contentHash)` ORDER BY path; content hash = deterministic SHA-256 hex (`node:crypto`), so a reopen manifest is a stable set comparison (SC-025) and byte-comparable across the Pg and in-memory stores.

## What Didn't Work / Gotchas
- **Uniform async reject channel:** store validation initially THREW synchronously, so Vitest `.rejects` never saw a promise. Fixed by making every failure a REJECTION — `Pg*` methods are `async`; the in-memory double returns `Promise.reject(...)`. Codify this for all future stores.
- **In-memory transaction modelling:** the in-memory `ProjectStore` models atomicity via `structuredClone` snapshot-and-restore-on-throw, plus a one-shot `failNextCommitAfter(n)` fault so SC-026 crash-atomicity is deterministic with no Postgres. The `pg-store.test.ts` (DATABASE_URL-gated) proves the REAL ROLLBACK by wrapping `Db` so the 2nd `project_file_versions` INSERT rejects.
- **Untrusted uuid `:id`:** `getProject` must catch Postgres `22P02` (invalid_text_representation) and return `null` — a non-uuid route param is "not found", not a 500. Only bites live SQL (in-memory ids aren't uuids), so it needs a pg-store assertion.
- **No config tunable owns the 30-day deletion-recovery window** (config's `versionRetentionDays` is the DIFFERENT D48 history window). Used a `deletionRecoveryDays` store option defaulting to `DEFAULT_DELETION_RECOVERY_DAYS = 30`. Candidate future tunable if D49's window must diverge.
