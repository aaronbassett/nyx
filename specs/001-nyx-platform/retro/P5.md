# Phase 5 Retrospective: User Story 2 — Compile pipeline

## What Worked Well
- **Reviewing the real `../compact-mcp` source BEFORE building** caught the spec-vs-reality gap (no R2/upload/urlPrefix) up front — constitution I paid off. Owner resolved it by introducing a **Compile Service** glue layer; contract authored + approved at `infra/compile-service/API.md` (committed, shareable with the service-building agent).
- **Contract-first, then build-against-it (mocked):** with the contract pinned, both the server client+orchestrator and the web fetch harness were built + fully green against mocks, deterministically, with zero real service/R2 — the real integration stays owner-gated (like real-Lace).

## Resolution & architecture (Compile Service glue)
- Nyx does NOT talk to `compact-mcp` directly and NEVER holds R2 write creds. A **Compile Service** sits between them: holds the only R2 write creds (constitution III), drives the MCP, uploads content-hashed artifacts, exposes an HTTP API Nyx consumes (`/v1/check`, `/v1/compile`+poll, `/v1/version`). Nyx authenticates with `COMPILE_SERVICE_TOKEN` and only ever READS R2.
- Content-addressed prefix `<projectId>/<sourceHash>/`; `sourceHash` folds in compilerVersion + flags so reuse (SC-006) can never serve keys from a different compiler.

## Patterns & Code
- **Compile client** (`apps/server/src/compile/client.ts`) mirrors the `wallet/auth.ts` injectable-HTTP pattern (server-side + required bearer). A **compile failure is DATA** (`ok:false` / job `status:failed` + diagnostics), never a throw; only transport faults / 4xx-5xx envelopes / malformed bodies / job-wait exhaustion throw. `runCompileJob` submits then polls to terminal with an injectable delay+clock + a bounded `maxWaitMs` (FR-016 — a hung job is an explicit timeout, never a loop).
- **Verify-before-announce** (`orchestrator.ts`): on `succeeded`, GET `<urlPrefix>/manifest.json` → zod-validate (§5) → HEAD every listed file → only then `emit(artifacts:ready)` once. Any gap → `verification-failed` + reopen guidance, NO emit. Orchestrator is stateless per turn, so at-most-once announce holds by construction.
- **Web artifact read** (`apps/web/src/artifacts/fetch.ts`): every artifact fetch uses `mode:"cors"` (R3: cors-mode is COEP-exempt, so it works under `require-corp`) + `credentials:"omit"` (no session cookie on the public R2 read domain, constitution III). `allOk` = zero 404s on a fresh prefix (SC-005/007); EC-10 oversized-uncached is orthogonal to `ok`.
- **Two distinct "manifest" concepts:** the VFS manifest `(path, contentHash)[]` (D38, `@nyx/protocol`) vs the R2 artifact `manifest.json` `{sourceHash, compilerVersion, circuits, files[]}` (§5, local types). Same word, different things — don't conflate.

## For Next Time (US1 wiring seam)
- **`COMPILE_SERVICE_TOKEN` + `COMPILE_SERVICE_URL` are NOT in `config/schema.ts` yet.** US2 has no consumer (the orchestrator takes injected deps), so this was deliberately deferred — US1 (which invokes the compile pipeline) must add them to `EnvSchema`/`Config` and construct `HttpCompileClient`. Flagged early so it isn't discovered late.
- Real Compile Service + R2 integration is **owner-gated** — validate the contract end-to-end once the service exists.

## ⚠️ BLOCKER — real compact-mcp diverges materially from the US2 spec (verified from `../compact-mcp` source, 2026-07-13)
The owner's toolchain MCP (`../compact-mcp`, Rust/rmcp) is a **local compile/analyze toolchain over a workspace directory** — it has NO R2, NO upload, NO urlPrefix, NO content-hash prefix, NO CDN concept (0 mentions in its design doc).

**Real tool contract:**
- `compile` `{path?|source? (XOR), target_dir?="build", skip_zk?=true, no_communications_commitment?=false, source_root?}` → `CompileOutcome {ok, proving_keys, diagnostics: Diagnostic[], duration_ms}` + `target_dir` (LOCAL path). Long builds → rmcp task (`execution(task_support="optional")`) + 2s progress-notification heartbeat + a serializing BuildGate + timeout. A compile FAILURE is a successful call with `isError:true` carrying the outcome; `McpError` only for the path/source XOR. `skip_zk:true` (default) = fast, no keys (emits contract-info.json/index.d.ts/.zkir); `skip_zk:false` = PLONK keys in `<target>/keys/`, zkir in `<target>/zkir/<circuit>.bzkir`.
- `diagnostics` `{path?|source?, max_diagnostics?}` → parser-only syntax diagnostics (ms, no compiler).
- `ast` / `symbols` / `stats`; `artifacts {target_dir}` / `zkir_stats {target_dir,circuit?}` / `witness_scaffold {path?|source?}`; `versions` (compactc/language/ledger/runtime/CLI/compactp + skew verdict — this is where the D6 compiler version lives, NOT in the compile response); `toolchain_list/check/update/clean`.
- `Diagnostic = {severity: error|warning|note, source: compactp|compactc, message, file?, span{start{offset?,line,column}, end?}, code?, raw}`.

**Divergences from the US2 spec (D6/D7/D12/D30/D31/D50):**
- Spec assumes the MCP "holds the only R2 write credentials," uploads prover/verifier/zkir/manifest to R2 under a content-hashed prefix, and returns `urlPrefix` + compiler version, with `artifacts:ready` after upload. **None of this exists in the MCP.** So the entire R2 artifact pipeline (FR-014, SC-005/006/007, D7, D12 `artifacts:ready`) has no home, and a zero-trust question is open (constitution III): if not the MCP, WHO holds R2 write creds and does the upload?
- File transfer unresolved: the MCP reads a filesystem workspace (path or inline single source); how do a Nyx project's Postgres-row files (US7) reach it, especially for multi-file imports?
- Compiler version (D6) is a separate `versions` call, not part of `compile`.

**→ Surfaced to owner; awaiting direction before implementing US2.**
